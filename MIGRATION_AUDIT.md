## 7. ARCHITECTURE ANALYSIS

### Current Architecture:
```
Browser ‚Üê‚Üí TanStack Start SSR (Vinxi/h3) ‚Üê‚Üí Supabase
                        ‚Üï
              Server Functions (createServerFn)
                        ‚Üï
            supabaseAdmin (service_role key)
```

### How Server Functions Work:
1. Client-side: `attachSupabaseAuth` middleware reads Supabase session token
2. RPC call sends Bearer token in header
3. Server-side: `requireSupabaseAuth` middleware validates JWT
4. Server function executes with `supabaseAdmin` (service role, bypasses RLS)
5. Returns data to client

This is actually a **good architecture** ‚Äî all sensitive auth and business logic is server-side. The challenge is migrating from TanStack Start to Next.js while preserving this pattern.

## 10. SECURITY FINDINGS

### Critical (Fix Immediately):
1. **None found** ‚Äî All sensitive logic is server-side, no service role key in client, no ZATCA secrets in client, no OTP logging.

### High:
1. **`.env` committed to repository** ‚Äî Contains Supabase URL and anon key. While anon key is client-safe, the `.env` should be `.gitignore`d and `.env.example` created.
2. **Frontend route guard only** ‚Äî The SPA router (`index.tsx`) has client-side guards but all actual enforcement is in server functions. When migrating to Next.js, protect `/dashboard/*` and `/pos/*` with middleware + server enforcement.

### Medium:
1. **`data.ts` contained mock data with hardcoded PINs** ‚Äî `CASHIERS` array previously had hardcoded PINs `1234` and `9999`. Since removed (Phase 5 security fix). Seed credentials now come from `NEXT_PUBLIC_SEED_*` env vars only.
2. **`PRODUCTS` array in `data.ts`** ‚Äî Static menu data used as fallback if Supabase is empty. Should be removed for production.
3. **`getModifierGroups` in `data.ts`** ‚Äî Uses static data, not database addon_groups. The POS screen uses this static data for modifier display, not the DB. Needs to be wired to DB.

### Low:
1. **`console.error` in production code** ‚Äî Multiple `console.error` calls for audit/ZATCA failures. Acceptable for now.
2. **Error messages expose "Connect Supabase in Lovable Cloud"** ‚Äî Lovable-specific error messages in Supabase client files.

## 11. BUGS AND RISKY AREAS

### Type/Build:
1. **Vite-specific imports** ‚Äî `import.meta.env.VITE_*` in `client.ts` won't work in Next.js without middleware.
2. **`crypto` import** ‚Äî `createHash` from `crypto` in `zatca.server.ts` is Node.js only. Works in TanStack Start SSR but need to ensure Next.js Web API compatibility.
3. **`@noble/curves` usage** ‚Äî Uses `Buffer.from()`, needs Node.js polyfill in edge runtime.

### Runtime:
1. **POS product modifiers use static data** ‚Äî The POS screen reads modifier options from `data.ts` (getModifierGroups), not from the DB `addon_groups` / `addon_groups` tables.
2. **Shift close timeout** ‚Äî `signOut` in `store.tsx` has a `window.confirm()` that may not work in all contexts.
3. **Routing is SPA-style** ‚Äî All screens rendered via switch-case in a single route. No real URL routing for deep linking.

### Migration Risks:
1. **TanStack Start Server Functions** must become Next.js Server Actions or Route Handlers ‚Äî large refactor of 15+ function files.
2. **RTL/LTR handling** ‚Äî Currently done via `useEffect` on `document.documentElement` in `store.tsx`. Next.js requires `next-intl` or similar for proper RTL.
3. **Tailwind CSS v4** ‚Äî Next.js currently works best with Tailwind v3 via PostCSS. Tailwind v4 has experimental Next.js support.
4. **React 19 compatibility** ‚Äî Next.js 15+ supports React 19, but some Radix UI primitives may have compatibility issues.
5. **42+ lib files** need to be adapted ‚Äî significant work but pattern is consistent.
